数据转换

  • CompressContent: 压缩解压内容
  • ConvertCharacterSet: 将用于将内容编码的字符集从一个字符集转换为另一个字符集
  • EncryptContent: 加密或解密内容
  • ReplaceText: 使用正则表达式修改文本内容
  • TransformXml: 对XML内容应用XSLT转换
  • JoltTransformJSON: 应用JOLT规范转换JSON内容

路由和中介

  • ControlRate: 控制数据流部分流量的速率
  • DetectDuplicate: 基于一些用户定义的标准，监控重复的流文件。通常与HashContent一起使用
  • DistributeLoad: 通过仅将部分数据分发给每个用户定义的关系来实现负载平衡或示例数据
  • MonitorActivity: 当用户定义的时间段过去而没有任何数据通过流中的特定点时，发送一个通知。当数据流恢复时，可以选择发送一个通知。
  • RouteOnAttribute: 根据它所包含的属性来路由FlowFile。
  • ScanAttribute: 扫描流文件中用户定义的属性集，检查属性是否与用户定义字典中的术语匹配。
  • RouteOnContent: 搜索流文件的内容，看看它是否匹配任何用户定义的正则表达式。如果是，则将FlowFile路由到配置的关系。
  • ScanContent: 在流文件的内容中搜索用户定义的字典中出现的术语，并根据这些术语的存在与否进行路由。字典可以由文本条目或二进制条目组成。
  • ValidateXml: 根据XML模式验证XML内容;根据用户定义的XML模式，根据流文件的内容是否有效来路由流文件。

访问数据库

  • ConvertJSONToSQL: 将JSON文档转换为SQL INSERT或UPDATE命令，然后将其传递给PutSQL处理器
  • ExecuteSQL: 执行用户定义的SQL SELECT命令，将结果以Avro格式写入一个流文件
  • PutSQL: 通过执行由FlowFile的内容定义的SQL DDM语句来更新数据库
  • SelectHiveQL: 对Apache Hive数据库执行用户定义的HiveQL SELECT命令，将结果以Avro或CSV格式写入流文件
  • PutHiveQL: 通过执行由FlowFile的内容定义的HiveQL DDM语句来更新Hive数据库

属性提取

  • EvaluateJsonPath: 用户提供JSONPath表达式(类似于XPath，用于XML解析/提取)，然后根据JSON内容对这些表达式进行评估，以替换FlowFile内容或将值提取到User -named属性中。
  • EvaluateXPath: 用户提供XPath表达式，然后根据XML内容对这些表达式进行评估，以替换FlowFile内容或将值提取到User -named属性中。
  • EvaluateXQuery: 用户提供一个XQuery查询，然后根据XML内容对该查询进行评估，以替换FlowFile内容或将值提取到User -named属性中。
  • ExtractText: User提供一个或多个正则表达式，然后根据FlowFile的文本内容对它们进行评估，然后将提取的值添加为用户命名属性。
  • HashAttribute: 对用户定义的现有属性列表的连接执行散列函数。
  • HashContent: 对流文件的内容执行散列函数，并将散列值作为属性添加。
  • IdentifyMimeType: 计算流文件的内容，以确定流文件封装的文件类型。这个处理器能够检测许多不同的MIME类型，例如图像、字处理器文档、文本和压缩格式等等。
  • UpdateAttribute: 向流文件添加或更新任意数量的用户定义属性。这对于添加静态配置的值以及使用表达式语言动态地派生属性值非常有用。该处理器还提供了一个“高级用户界面”，允许用户根据用户提供的规则有条件地更新属性。

系统交互

  • ExecuteProcess: 运行用户定义的操作系统命令。流程的标准输出被重定向，以便写入标准输出的内容成为出站流文件的内容。这个处理器是一个源处理器——它的输出将生成一个新的流文件，而系统调用将不会接收任何输入。为了向流程提供输入，请使用ExecuteStreamCommand处理器。
  • ExecuteStreamCommand: 运行用户定义的操作系统命令。FlowFile的内容可选地流到进程的StdIn。写入StdOut的内容成为hte出站流文件的内容。这个处理器不能作为源处理器使用——它必须被输入流文件来执行它的工作。要使用源处理器执行相同类型的功能，请参阅ExecuteProcess处理器。

数据摄取

  • GetFile: 将文件的内容从本地磁盘(或网络附加磁盘)流到NiFi中，然后删除原始文件。此处理器预期将文件从一个位置移动到另一个位置，不用于复制数据。
  • GetFTP: 通过FTP将远程文件的内容下载到NiFi中，然后删除原始文件。此处理器预期将数据从一个位置移动到另一个位置，不用于复制数据。
  • GetSFTP: 通过SFTP将远程文件的内容下载到NiFi中，然后删除原始文件。此处理器预期将数据从一个位置移动到另一个位置，不用于复制数据。
  • GetJMSQueue: 从JMS队列下载消息并基于JMS消息的内容创建流文件。JMS属性也可以作为属性复制。
  • GetJMSTopic: 从JMS主题下载消息并基于JMS消息的内容创建流文件。JMS属性也可以作为属性复制。该处理器支持持久订阅和非持久订阅。
  • GetHTTP: 将远程HTTP或基于HTTP的URL的内容下载到NiFi中。处理器将记住ETag和最后修改日期，以确保数据不会被不断地摄取。
  • ListenHTTP: 启动HTTP(或HTTPS)服务器并侦听传入的连接。对于任何传入的POST请求，都将请求的内容作为一个流文件写入，并返回一个200响应。
  • ListenUDP: 侦听传入的UDP数据包，并为每个数据包或每个包包包包创建一个流文件(取决于配置)，然后将流文件发送到“成功”关系。
  • GetHDFS: 监视HDFS中用户指定的目录。每当一个新文件进入HDFS时，它都会被复制到NiFi并从HDFS中删除。此处理器预期将文件从一个位置移动到另一个位置，不用于复制数据。如果在集群中运行，则此处理器也只能在主节点上运行。为了从HDFS中复制数据并保留它，或者为了从集群中的多个节点中传输数据，请参见ListHDFS处理器。
  • ListHDFS / FetchHDFS: ListHDFS监视HDFS中用户指定的目录，并发出一个流文件，其中包含遇到的每个文件的文件名。然后，它通过分布式缓存在整个NiFi集群中保持这种状态。然后可以在集群中展开这些流文件，并将其发送到FetchHDFS处理器，该处理器负责获取这些文件的实际内容，并发出包含从HDFS获取的内容的流文件。
  • FetchS3Object: 从Amazon Web Services (AWS)简单存储服务(S3)获取对象的内容。出站流文件包含从S3接收的内容。
  • GetKafka: 从Apache Kafka获取消息，特别是针对0.8。x版本。这些消息可以作为每个消息的流文件发出，也可以使用用户指定的分隔符组合在一起。
  • GetMongo: 对MongoDB执行用户指定的查询，并将内容写入新的流文件。
  • GetTwitter: 允许用户注册一个过滤器来监听Twitter的“花园软管”或企业端点，为收到的每个tweet创建一个流文件。

数据输出/发送数据

  • PutEmail: 向配置的收件人发送电子邮件。FlowFile的内容可以选择作为附件发送。
  • PutFile: 将流文件的内容写入本地(或网络附加)文件系统上的目录。
  • PutFTP: 将流文件的内容复制到远程FTP服务器。
  • PutSFTP: 将流文件的内容复制到远程SFTP服务器。
  • PutJMS: 将流文件的内容作为JMS消息发送到JMS代理，可以根据属性添加JMS属性。
  • PutSQL: 以SQL DDL语句(插入、更新或删除)的形式执行流文件的内容。FlowFile的内容必须是有效的SQL语句。可以将属性用作参数，以便将流文件的内容参数化SQL语句，以避免SQL注入攻击。
  • PutKafka: 将流文件的内容作为消息发送给Apache Kafka，特别是针对0.8。x版本。可以将FlowFile作为单个消息或分隔符发送，例如可以指定新行，以便为单个FlowFile发送多个消息。
  • PutMongo: 将流文件的内容作为插入或更新发送到Mongo。

分裂和聚合

  • SplitText: SplitText接受一个内容为文本的流文件，并根据配置的行数将其分割为一个或多个流文件。例如，可以将处理器配置为将一个流文件分割成多个流文件，每个流文件只有一行。
  • SplitJson: 允许用户将由数组或多个子对象组成的JSON对象拆分为每个JSON元素的一个流文件。
  • SplitXml: 允许用户将XML消息分割成多个流文件，每个流文件包含原始流的一部分。这通常在几个XML元素与一个“包装器”元素结合在一起时使用。然后，该处理器允许将这些元素拆分为单独的XML元素。
  • UnpackContent: 解压缩不同类型的归档格式，如ZIP和TAR。归档文件中的每个文件随后作为单个流文件传输。
  • MergeContent: 这个处理器负责将多个流文件合并到一个流文件中。可以通过将它们的内容与可选的页眉、页脚和demarcator连接在一起，或者通过指定归档格式(如ZIP或TAR)来合并流文件。可以根据公共属性将流文件绑定在一起，或者如果它们被其他一些分割过程分割，则可以进行“碎片整理”。每个容器的最小和最大大小是用户指定的，基于元素的数量或flowfile内容的总大小，还可以分配一个可选的超时，以便FlowFiles只会等待它们的容器在一定时间内变得满。
  • SegmentContent: 根据配置的数据大小，将一个流文件分割成许多更小的流文件。拆分不是针对任何类型的demarcator执行的，而是基于字节偏移量执行的。这是在传输流文件之前使用的，以便通过并行发送许多不同的块来提供更低的延迟。另一方面，MergeContent处理器可以使用碎片整理模式重新组装这些流文件。
  • SplitContent: 将一个流文件分割成多个流文件，类似于SegmentContent。但是，对于SplitContent，不是在任意的字节边界上执行拆分，而是指定一个字节序列来拆分内容。

HTTP

  • GetHTTP: 将远程HTTP或基于HTTP的URL的内容下载到NiFi中。处理器将记住ETag和最后修改日期，以确保数据不会被不断地摄取。
  • ListenHTTP: 启动HTTP(或HTTPS)服务器并侦听传入的连接。对于任何传入的POST请求，都将请求的内容作为一个流文件写入，并返回一个200响应。
  • InvokeHTTP: 执行由用户配置的HTTP请求。这个处理器比GetHTTP和PostHTTP更加通用，但是需要更多的配置。这个处理器不能用作源处理器，需要有传入的流文件来触发它来执行它的任务。
  • PostHTTP: 执行HTTP POST请求，将流文件的内容作为消息体发送。这通常与ListenHTTP一起使用，以便在无法使用站点到站点的情况下(例如，当节点之间不能直接访问并且能够通过HTTP代理进行通信时)在两个不同的NiFi实例之间传输数据。注意:除了现有的原始套接字传输之外，HTTP还可以作为站点到站点的传输协议使用。它还支持HTTP代理。建议使用HTTP站点到站点，因为它具有更好的可伸缩性，并且可以使用具有更好的用户身份验证和授权的输入/输出端口提供双向数据传输。
  • HandleHttpRequest / HandleHttpResponse: HandleHttpRequest处理器是一个源处理器，它启动一个嵌入式HTTP(S)服务器，类似于ListenHTTP。但是，它不会向客户机发送响应。相反，FlowFile被发送出去，HTTP请求的主体作为它的内容，所有典型的Servlet参数、头等的属性作为属性。然后，HandleHttpResponse能够在流程文件完成处理后将响应发送回客户端。这些处理器总是被期望与其他处理器一起使用，并允许用户在NiFi中可视化地创建Web服务。这对于向非基于web的协议添加前端或围绕NiFi已经执行的某些功能(如数据格式转换)添加简单的web服务尤其有用。

Amazon Web Services

  • FetchS3Object: 获取存储在Amazon Simple Storage Service (S3)中的对象的内容。从S3检索的内容然后写入到流文件的内容。
  • PutS3Object: 使用配置的凭据、密钥和bucket名称将流文件的内容写入Amazon S3对象。
  • PutSNS: 将流文件的内容作为通知发送到Amazon Simple notification Service (SNS)。
  • GetSQS: 从Amazon Simple queue Service (SQS)获取一条消息，并将消息的内容写入到流文件的内容中。
  • PutSQS: 将流文件的内容作为消息发送给Amazon简单排队服务(SQS)。
  • DeleteSQS: 从Amazon简单排队服务(SQS)中删除一条消息。这可以与GetSQS一起使用，以便接收来自SQS的消息，对其执行一些处理，然后在它成功完成处理后从队列中删除对象。
